# 1. Введение
## 1.1. Назначение

Полное название: **Спецификация требований (SRS) к приложению "К столу!"**  
Версия документа: v0.1 (черновик)  
Дата последней редакции: 13.12.2025  
Продукт: Release 0.1 (MVP)  

Данный документ описывает функциональные и нефункциональные требования к приложению "К столу".  

Аудитория документа и цель использования документа:
| Аудитория | Цель |
| --- | --- |
| Владелец продукта | Согласовать объём MVP |
| Системный аналитик | Вести и поддерживать требования |
| Разработчик | Понять, что реализовать и границы проекта |
| Тестировщик | Получить основу для плана тестирования |
| UI/UX дизайнер | Понимать сценарии использования |
| Ревьюеры | Оценить полноту и реализуемость требований |

Данный документ используется как базовый источник требований при разработке, тестировании и приёмке MVP.  


## 1.2. Соглашения, принятые в документах

Термины и определения в документе используются в значениях, приведённых в Приложении А (Глоссарий).

Термины из глоссария в тексте выделяются ***полужирным курсивом***. Далее по тексту термин используется без выделения.

Пользовательские проблемы (***User Problems***) имеют идентификатор вида UP-<номер подсистемы><номер> и служат основным источником требований, связанных с пользовательскими задачами.  

Пользовательские истории (***User Stories***) имеют идентификатор US-<подсистема (char)><номер> и описывают ценность для пользователя.  

Варианты использования (***Use Cases***) имеют идентификатор UC-<подсистема (char)><раздел подсистемы><номер> и служат для отображения взаимодействий пользователя и системы. Могут быть связаны с пользовательскими историями. 

Бизнес-правила (***Business Rules***) имеют идентификатор вида BR-<подсистема (char)><раздел подсистемы><номер> и задают ограничения и правила работы системы.  

Функциональные требования (***Functional requirements***) имеют идентификатор вида FR-<подсистема (char)><раздел подсистемы><номер> и задают функциональность, которую система должна реализовать.

Нефункциональные требования (***Non-Functional requirements***) имеют идентификатор вида **TBD** и задают критерии качества работы системы.

Диаграмма трассировки требований представлена на рисунке [1.1](/Приложение%20Б.%20Диаграммы/Диаграмма%201.1%20-%20%20Трассировка%20требований.png). TBD (включить в схему связь предположения -> NFR?)

Можно проследить функциональные требования берут своё начало от ***Use Case*** или ***User Story***, ***Use Case*** может быть связан с ***User Story*** и пользовательская история прямо или косвенно решает одну из проблем пользователя (***User Problem***).

Приоритеты требований задаются тремя уровнями: высокий, средний, низкий и указываются как атрибут для FR/NFR.

Единицы измерения и правила агрегации ингредиентов определены в разделе 4 и бизнес-правилах.  

Все требования в документе формулируются в виде утверждений с модальностью «должна/должен» (например: «Система должна …»).

## 1.3. Границы проекта

### 1.3.1. Общее описание проекта 

Приложение "К столу!" предназначено для облегчения процесса готовки блюд к мероприятиям - от планирования до приготовления. Оно предоставляет единое место хранения всех рецептов, автоматическое создание списка покупок и возможность совместной работы. Совместная работа предлагает общий доступ к рецептам и общую отметку купленных продуктов.

Общий документ видения и границ представлен [тут](../vision%20and%20scope.md).

Данный SRS описывает требования Release 0.1 (MVP).

### 1.3.2. Область охвата Release 0.1 (MVP)

В рамках Release 0.1 (MVP) разрабатывается веб-версия приложения и реализуется следующий функционал:

1. **Рецепты**
   1.1. Пользователь может вести личную библиотеку **_Рецептов_** (просмотр списка рецептов).
   1.2. Пользователь может добавить **_Рецепт_** вручную.
   1.3. Пользователь может редактировать и удалять свои **_Рецепты_**.

2. **Мероприятия**
   2.1. Пользователь может создать **_Мероприятие_**.
   2.2. Пользователь может добавить в **_Мероприятие_** **_Рецепты_** из своей библиотеки (и удалить их из мероприятия при необходимости).
   2.3. Пользователь может просматривать список своих мероприятий.

3. **Список покупок**
   3.1. Система формирует **_Список покупок_** на основе рецептов **_Мероприятия_**.
   3.2. Пользователь может просматривать **_Список покупок_** и отмечать купленные позиции.

4. **Совместная работа**
   4.1. Система предоставляет доступ к **_Мероприятию_** по ссылке без регистрации (роль **_Участник по ссылке_**).
   4.2. **_Организатор_** копирует ссылку-приглашение и отправляет её вне приложения.
   4.3. **_Участник по ссылке_** может просматривать список рецептов **_Мероприятия_** без возможности редактирования и работать со **_Списком покупок_** (отмечать купленные позиции).


### 1.3.3. Вне области охвата Release 0.1 (MVP)

В рамках Release 0.1 (MVP) **не реализуются** следующие возможности:

- Архивирование **_Рецептов_** и расширенные механизмы организации библиотеки (теги, папки, избранное, поиск с фильтрами).
- Полноценная авторизация и управление аккаунтами пользователей (регистрация, вход, восстановление доступа, профили).
- Роли и права, отличные от **_Организатора_** и **_Участника по ссылке_** (например, «просмотр рецептов без доступа к списку покупок», назначение прав конкретным участникам).
- Управление участниками внутри приложения (добавление/удаление участников, список участников мероприятия, приглашения “внутри” приложения).
- Обмен **_Рецептами_** между пользователями (публичные рецепты, общий каталог, отправка/копирование рецепта другому пользователю).
- Автоматический импорт **_Рецептов_** из внешних источников (сайты, файлы, распознавание текста и т.п.).
- Расширенная работа с ингредиентами и единицами измерения: приведение количества ингредиентов к единой системе при агрегировании, включая конвертацию между несовместимыми типами единиц (например, «шт» → «г»), а также пользовательская смена единиц отображения в **_Списке покупок_**.
- Уведомления и напоминания (push/e-mail/мессенджеры), а также интеграции с календарём.
- Интеграции с магазинами/доставкой/ценами (подбор товаров, расчёт стоимости, сравнение цен, оформление заказа).
- Оффлайн-режим и гарантированная синхронизация при нестабильном интернете.
- Встроенный чат/комментарии/обсуждения внутри **_Мероприятия_**.
- Вложения и медиа (фото рецептов, сканы, файлы) и их хранение.
- Нативные мобильные приложения (Android/iOS) и отдельная десктоп-версия.

### 1.3.4. Границы ответственности системы и внешние зависимости

#### Границы ответственности системы

В рамках Release 0.1 (MVP) ответственность **_Приложения «К столу!»_** включает:
- предоставление веб-интерфейса для работы **_Организатора_** и **_Участника по ссылке_**;
- хранение данных **_Рецептов_**, **_Мероприятий_** и **_Списков покупок_**;
- формирование **_Списка покупок_** на основе рецептов, добавленных в **_Мероприятие_**;
- обеспечение совместного доступа к данным **_Мероприятия_** по **_Ссылке-приглашению_** в рамках прав роли **_Участник по ссылке_**;
- синхронизацию изменений, связанных с отметкой купленных позиций в **_Списке покупок_**.

#### Внешние зависимости

Работа системы зависит от следующих внешних факторов и компонентов:
- устройство пользователя с современным веб-браузером и доступом в интернет;
- сетевое соединение между клиентом (браузером) и серверной частью приложения;
- инфраструктура размещения (хостинг/сервер/БД), обеспечивающая доступность веб-приложения;
- механизмы хранения данных на стороне клиента (например, локальное хранилище браузера) для идентификации **_Организатора_** в рамках упрощённой модели доступа MVP;
- каналы связи вне приложения (мессенджеры, e-mail и т.п.), через которые **_Организатор_** отправляет **_Ссылку-приглашение_**.

#### Вне ответственности системы

В рамках Release 0.1 (MVP) **_Приложение «К столу!»_** не отвечает за:
- доставку/покупку продуктов и действия пользователей в реальном мире;
- работу сторонних приложений и сервисов, используемых для передачи **_Ссылки-приглашения_** (мессенджеры, почта и др.);
- содержание и доступность внешних источников рецептов (сайты/сервисы), а также корректность данных, находящихся вне приложения;
- работоспособность системы при отсутствии интернет-соединения или при блокировках/ограничениях на стороне сетевой инфраструктуры пользователя.


### 1.3.5. Потенциальные расширения

Ниже перечислены возможные направления развития продукта в последующих релизах. Данный список носит справочный характер и **не является** требованиями Release 0.1 (MVP):

- Полноценная авторизация и аккаунты пользователей (регистрация, вход, восстановление доступа), а также управление профилями.
- Расширенная модель ролей и прав доступа: **_Участник_** (с аккаунтом), просмотр рецептов без доступа к **_Списку покупок_**, назначение прав конкретным участникам **_Мероприятия_**.
- Управление участниками **_Мероприятия_** внутри приложения: список участников, приглашения, удаление участников, управление доступом.
- Расширенная организация библиотеки **_Рецептов_**: архивирование, избранное, теги, папки, поиск и фильтры.
- Расширенная работа с ингредиентами и единицами измерения: конвертация между типами единиц (например, «шт» ↔ «г»), пользовательская смена единиц отображения в **_Списке покупок_**, справочник ингредиентов и пользовательские настройки.
- Импорт **_Рецептов_** из внешних источников (по ссылке/тексту/файлу) и инструменты ускоренного ввода.
- Обмен **_Рецептами_** между пользователями и/или публичный каталог рецептов.
- Уведомления и напоминания, интеграции с календарём.
- Интеграции с магазинами/доставкой и оценка стоимости **_Списка покупок_**.
- Поддержка нативных мобильных приложений (Android/iOS) и расширенные оффлайн-сценарии.
